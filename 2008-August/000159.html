> where actionset is a list of actions that belong together. E.g.:
> - a Keypress actionset with 52 actions (not much benefit there compared

> - a MPlayer actionset with actions Play, Pause, Stop, Next, Prev, etc

these are both bad examples. The mplayer Module is currently a thin
wrapper around the mplayer remote control syntax (Have a look at
"mplayer -input cmdlist"). This is as easy as it can get.

We already have a place where "knowing" people can create compounds and
new modules. These places are the modules. From a module you have the
full power of python at your finger tips. If the config is really that
difficult, you should hide it alltogether.

This leaves complex dbus calls and complex commandlines to the action
list. And these could also be summarised in a module (Lets call it
'Little helpers at your fingertips'.

Thinking again about this, you could add a comment field and shortname
for each action, which could provide Joe-Normal-User a hit whats going
on. Add in an "Action library" from which you can copy'n'past or
drag'n'drop and you're there. The library contains predefined handler
calls with the above mentioned comments and shortnames. Additionally you
can create a hierarchy and so on. But there is no connection between
library and the handlers, after the copying the action, it is part of
the profile and can be modified seperately.
