Am Dienstag, den 19.08.2008, 23:10 +0200 schrieb Jasper Groenewegen:
> > This basicly exists. A Scheme is just a more intelligent module. Let's
> > go back to the Player example. Currently the Rhythmbox Profile uses the
> > DBUS-Module. But it does not have to. You could also programm a
> > Rhythmbox Module, that hides the whole DBUS handling and just offers:
> > play, forward and so on.
> Ah, this brings me to an issue with the current design: Currently, a key
> press/release is registered and all event handlers are polled to see if
> they map this keycode in the active profile. This allows for
> flexibility, I guess this way 1 keypress could mean 'left mouse button'
> 'Rhythmbox.play' and 'Walk the dog' at the same time, but in my opinion
> 1 keypress should in principle be mapped to one action. Note that it is
> still possible to create an action that consists of a set of calls/input
> events.

You get inflexible this way. Think about the presentation module. There
are many actions when pressing the next slide key: restart the slide
timer, save the record for the last slide and switch to next slide.

The handling of the slide timer and records is pretty easy (one module
and your done), but there are n+1 programms to show presentations, each
with slithly different keybindings. Why make it unnecessry complex and
just provide a way to send request via dbus, keyinput and so on.

I see the point for the end user. But the multiple programms can easily
tangled by providing pre created profiles für acroread, evince, OOo
Impress and so on. This coupled with a new config inteface shoudl do
it. 

What I think of is this:

Make the tabs in the profile section "Init", "Actions", "Destruct" or
their corresponding title for "normal" people. Inside these you find a
treeview, with keycode (on the phone, if you want also with Alias Names)
and the associated actions. The actions are short text desciptions and a
double click opens a window, where you can configure the action. Action
in this case means a combination of handler and config for it.

This frontend config can be decomposed to the already known config
format. And the current config format can be put into the format
described above. By treeview reordering you could sort by handler,
key/keycode and paramters (this would be stupid).

I tried to do a GUI Mockup - I thing it shows my idea. If a screen
designer wants to step up to improve I have no problem with that.
