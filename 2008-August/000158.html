> I'm thinking about the situation where a whole keyboard gets mapped. The
> actionslist is then least 52 entries long. With action names like
> "keypress_a" or "keypress_A". Not really helpfull.
He he, in that way it would not be very helpfull no. But, it does have
the advantage that you can map the action once and reuse multiple times
in the profile. But I see the problem now that if there are 52 keypress
actions + 10 MPlayer actions + 6 evince actions + 8 OOimpress actions,
etc, the list to scroll to find the right action to map becomes very
very long. So perhaps it does have to be a 3 column list for Profile:
profile -> {keycode->actionset->action}
where actionset is a list of actions that belong together. E.g.:
- a Keypress actionset with 52 actions (not much benefit there compared
to bare handler/parameter setting)
- a MPlayer actionset with actions Play, Pause, Stop, Next, Prev, etc
which would be less frightening to Joe compared to the bare
handler/parameter list.

The action tab in this case would be a top row similar to the profile
top row (dropdown list + 3 buttons [new, edit, delete]) but with action
sets and the bulk of the action tab would be very similar to what the
profile tab currently is, but instead of the first column being keycode
it could be handler to eliminate the current inner tabs. The action set
wouldn't need 'on start'/'on stop' event definitions I think.
