Hey,

so I try to summarise:

This is the current situation:

profile -> {keycode->handler/parameter}

A profile is a mapping between a keycode and an associated
handler/parameter.

Your Idea sounds like this:

profile -> {keycode->action}

and

action -> [handler/paramater]

A profile is a mapping between keycode and associated action and an
action is a mapped to one or more handler/paramters.

I'm thinking about the situation where a whole keyboard gets mapped. The
actionslist is then least 52 entries long. With action names like
"keypress_a" or "keypress_A". Not really helpfull.

I see benefits, if you define actions based on the dbus and
commandexecutor module. The example was for a media player. But then a
mediaplayer might want to keep an internal state or needs more work
under the hood (like the mplayer module). If you go that far you can
create a generic Player module, which specific player is set at profile
selection and that "knows" how to handle each player. This is currently
done in the VolumeControl Module, that has currently support for alsa
and oss (pulseaudio needs work, as there are no channels, but instead
each sind is identified as a device on a server).
